1. ゴールと設計方針

1-1. プロジェクトのゴール

  • 犬OK宿の検索メディア「犬旅日和」を、SEOで上位を獲得できる構造に改修する
  • 週1回の自動更新で新規宿を発見し、人手は最終承認だけに絞る
  • エンジニア1名＋編集1名の2名体制でも回る運用設計
  • 5月8日の社内ミーティングまでに方針を固め、第1スプリント（DB＋移行＋エージェントA最小実装）まで着手する

1-2. 設計の根本方針

原則具体
シンプル最優先ライブラリは最小。Prisma併用なし、Supabaseに一本化。複雑性は運用工数になる。
AIが書きやすい構造正規化テーブル、UNIQUE制約による冪等性、status列で人間レビューを強制。
追跡可能性すべてのレコードに「いつ・誰（or どのエージェント）が・どこから取得したか」を記録。
段階的移行microCMSは即時廃止せず、Supabaseに移行後しばらく並行稼働。読み出しはSupabase優先で切り替え。
SEO非破壊既存URL /hotel/[id] から /hotel/[slug] への切替は301で連続性を保つ。

3. システム全体図

読み取り経路（ユーザー）と書き込み経路（自動更新）を分けた構成。

┌─────────────────────────── ユーザー（読み取り） ───────────────────────────┐ │ │ │ ブラウザ ─→ Vercel (Next.js / ISR) │ │ │ │ │ ├─→ Supabase PostgREST (一覧・詳細・検索) │ │ └─→ Supabase Storage (画像) │ │ │ └────────────────────────────────────────────────────────────────────────────┘ ┌──────────────────── 自動更新（週1 / Slackコマンド） ────────────────────────┐ │ │ │ ① 週1 cron (GitHub Actions schedule) │ │ ↓ │ │ エージェントA: 楽天API + じゃらんスクレイピング │ │ ↓ │ │ 重複判定 (dedup_hash) → discovery_queue へINSERT │ │ ↓ │ │ Slack通知 (Block Kit: 追加 / スキップ ボタン) │ │ │ │ ② Slack「@犬旅bot 〇〇宿を追加」 │ │ ↓ │ │ Vercel /api/slack/events ── 3秒以内に200 ─→ GitHub repository_dispatch │ │ ↓ │ │ エージェントB: 詳細スクレイピング → hotels INSERT (status=draft) │ │ ↓ │ │ Slack通知 → 編集者がSupabase Studioで承認 → status=published │ │ │ └────────────────────────────────────────────────────────────────────────────┘
重要: ユーザー読み取り経路と自動更新経路は完全に分離。読み取りはRLSで status='published' のみ公開、書き込みはservice_roleで行う。

-- 拡張

CREATE EXTENSION IF NOT EXISTS pg_trgm;       -- 部分一致検索（日本語OK）
CREATE EXTENSION IF NOT EXISTS postgis;       -- 地理検索

-- ENUMs

  'glamping', 'minshuku', 'resort', 'other'

-- マスタ

-- ========================================
-- hotels（マスター）
-- ========================================
CREATE TABLE hotels (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  prefecture_id   uuid NOT NULL REFERENCES prefectures(id),
  name            varchar(200) NOT NULL,

-- インデックス
CREATE INDEX idx_hotels_prefecture ON hotels(prefecture_id);
CREATE INDEX idx_hotels_status     ON hotels(status) WHERE status = 'published';
CREATE INDEX idx_hotels_geo        ON hotels USING GIST(geo);
-- 部分一致検索（v1のtsvector('japanese')はPostgresに存在しないためtrgmで代替）
CREATE INDEX idx_hotels_name_trgm  ON hotels USING GIN(name gin_trgm_ops);
CREATE INDEX idx_hotels_addr_trgm  ON hotels USING GIN(address gin_trgm_ops);

-- lat/lng → geo を自動同期するトリガー
CREATE OR REPLACE FUNCTION sync_hotel_geo() RETURNS trigger AS $$
BEGIN
  IF NEW.lat IS NOT NULL AND NEW.lng IS NOT NULL THEN
    NEW.geo := ST_SetSRID(ST_MakePoint(NEW.lng, NEW.lat), 4326)::geography;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER hotels_sync_geo
  BEFORE INSERT OR UPDATE OF lat, lng ON hotels
  FOR EACH ROW EXECUTE FUNCTION sync_hotel_geo();

-- updated_at 自動更新
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS trigger AS $$
BEGIN NEW.updated_at := now(); RETURN NEW; END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER hotels_updated_at
  BEFORE UPDATE ON hotels
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE INDEX idx_revisions_hotel ON hotel_revisions(hotel_id, created_at DESC);

-- agent_runs（エージェント実行ログ）★新規
-- ========================================
CREATE TABLE agent_runs (
  id            bigserial PRIMARY KEY,
  agent_name    varchar(50) NOT NULL,        -- agent_a / agent_b
  trigger_type  varchar(20) NOT NULL,        -- schedule / slack / manual
  started_at    timestamptz DEFAULT now(),
  finished_at   timestamptz,
  status        varchar(20),                  -- running / success / error
  stats         jsonb,                            -- { fetched: 80, new: 5, dup: 75 }

);

-- RLS（公開ポリシー）
-- ========================================
ALTER TABLE hotels             ENABLE ROW LEVEL SECURITY;
ALTER TABLE hotel_dog_policies ENABLE ROW LEVEL SECURITY;

-- 公開: status=published のみ anon で読める
CREATE POLICY hotels_public_read ON hotels
  FOR SELECT TO anon
  USING (status = 'published');

-- 子テーブルは親(hotel)が公開なら公開
CREATE POLICY dog_policies_public_read ON hotel_dog_policies
  FOR SELECT TO anon
  USING (EXISTS (SELECT 1 FROM hotels h WHERE h.id = hotel_id AND h.status = 'published'));

-- discovery_queue, hotel_revisions, hotel_sources, agent_runs は anon 不可（service_roleのみ）

7. 自動更新フロー設計 ⭐

7-1. エージェントA — 週次新規発見

項目内容
起動GitHub Actions schedule: cron: "0 0 * * 1"（毎週月曜9時 JST = 月曜0時 UTC）
入力都道府県47件 × 「ペット可」フィルタ
処理(1) 楽天トラベルAPIで一覧取得 → (2) じゃらんスクレイピング（Playwright）→ (3) 結果を正規化 → (4) dedup_hash 算出 → (5) 既存 hotels または discovery_queue に同ハッシュがあればスキップ → (6) 新規分のみ discovery_queue に INSERT
出力Slack #inutabi-discovery に「今週の新規候補 N件」サマリ＋各候補ごとに Block Kit メッセージ（追加/スキップ ボタン付き）
失敗時agent_runs.status = 'error' で記録、Slackにエラー通知

7-2. エージェントB — Slackからの個別追加・詳細補強

項目内容
起動(1) Slackボタン「追加」押下 / (2) Slackメンション「@犬旅bot 〇〇宿を追加」/ (3) 既存宿の詳細リフレッシュ
処理(1) discovery_queue または ユーザー指定URLから対象を特定 → (2) 楽天/じゃらん/公式サイトから詳細取得 → (3) 犬条件を抽出（部屋OK/サイズ/料金/ドッグラン等）→ (4) hotels に status='reviewing' でINSERT → (5) hotel_sources, hotel_dog_policies, hotel_images に関連レコード → (6) hotel_revisions に変更履歴
出力Slackに「✅ 〇〇宿を reviewing で登録しました。Supabase Studioで確認 → 公開」リンク付き

7-4. Slack ↔ Vercel ↔ GitHub Actions の3秒問題

制約: Slackは Webhook受信後3秒以内に200を返さないとタイムアウト扱いになる。スクレイピングは数十秒〜数分かかるため、Vercelで処理を完結させてはいけない。

解決策:

  1. Vercel /api/slack/events は受信→署名検証→GitHub repository_dispatch APIをfire-and-forget→200を即返す（<1秒）
  2. GitHub Actionsの workflow_dispatch受け取り側でエージェントBを起動（数分かかってOK）
  3. 完了したら Slack chat.postMessage で結果を投稿

7-6. 重複判定ロジック

// scripts/agents/lib/dedup.ts
import { createHash } from 'node:crypto';
import { normalizeAddress, normalizeHotelName } from './normalize';

export function computeDedupHash(input: {
  name: string;
  address: string;
  phone?: string;
}): string {
  const name = normalizeHotelName(input.name);  // 株式会社/ホテル/HOTEL等を除去
  const addr = normalizeAddress(input.address); // 全角半角・丁目番地統一
  const tail = (input.phone ?? '').replace(/\D/g, '').slice(-4);
  return createHash('sha256').update(`${name}|${addr}|${tail}`).digest('hex');
}

// src/lib/queries/hotels.ts
import { supabase } from '@/lib/supabase/client';

export async function getHotelBySlug(slug: string) {
  const { data, error } = await supabase
    .from('hotels')
    .select(`
      *,
      prefecture:prefectures(*, region:regions(*)),
      dog_policy:hotel_dog_policies(*),
      hotel_amenities(amenity:amenities(*)),
      hotel_images(*),
      booking_links(*),
      hotel_faqs(*)
    `)
    .eq('slug', slug)
    .eq('status', 'published')
    .maybeSingle();   // ← .single() ではなく maybeSingle() で404を素直に扱う

  if (error) throw error;
  if (!data) return null;

  // ★v1のバグ: 1:1リレーションでも配列で返ってくる場合がある
  // hotel_dog_policies はPK=hotel_idなので Supabase はオブジェクトで返すが、
  // 念のため両対応にしておく
  const dogPolicy = Array.isArray(data.dog_policy) ? data.dog_policy[0] : data.dog_policy;
  return { ...data, dog_policy: dogPolicy };
}

// メイン画像だけ欲しい場合は子テーブルにフィルタを掛ける別クエリ
// （v1の `images:hotel_images(url).filter(is_main.eq.true)` は存在しない構文）
export async function getHotelMainImage(hotelId: string) {
  const { data } = await supabase
    .from('hotel_images')
    .select('storage_path, alt_text')
    .eq('hotel_id', hotelId)
    .eq('is_main', true)
    .maybeSingle();
  return data;
}

// 一覧でメイン画像も併せて欲しい場合は !inner で必須結合
export async function listHotelsWithMainImage(prefectureSlug: string) {
  const { data: pref } = await supabase
    .from('prefectures').select('id').eq('slug', prefectureSlug).maybeSingle();
  if (!pref) return [];

  const { data } = await supabase
    .from('hotels')
    .select('*, hotel_images!inner(storage_path, alt_text)')
    .eq('prefecture_id', pref.id)
    .eq('status', 'published')
    .eq('hotel_images.is_main', true);   // ← 子テーブルへのフィルタはこの形
  return data ?? [];
}

8-4. Slack Events エンドポイント（3秒以内応答）

// src/app/api/slack/events/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { verifySlackSignature } from '@/lib/slack/sign';

export async function POST(req: NextRequest) {
  const raw = await req.text();
  if (!verifySlackSignature(req.headers, raw)) {
    return new NextResponse('unauthorized', { status: 401 });
  }

  const body = JSON.parse(raw);

  // SlackのURL検証チャレンジ
  if (body.type === 'url_verification') {
    return NextResponse.json({ challenge: body.challenge });
  }

  // app_mention → エージェントBをGitHub Actionsで起動（fire and forget）
  if (body.event?.type === 'app_mention') {
    const text = body.event.text as string;
    // 「@犬旅bot 〇〇宿を追加 https://...」のようなパターン
    const url = text.match(/https?:\/\/\S+/)?.[0];

    // awaitせずfire-and-forget。3秒以内に200を返す。
    fetch(`https://api.github.com/repos/${process.env.GH_OWNER}/${process.env.GH_REPO}/dispatches`, {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${process.env.GH_DISPATCH_TOKEN}`,
        Accept: 'application/vnd.github+json',
      },
      body: JSON.stringify({
        event_type: 'enrich-hotel',
        client_payload: { url, thread_ts: body.event.ts, channel: body.event.channel },
      }),
    }).catch(console.error);
  }

  return NextResponse.json({ ok: true });
}

9. URL移行計画（[id] → [slug]）

現状 /hotel/[id] で公開されているページから、SEOフレンドリーな /hotel/[slug] へ移行する。検索インデックスの連続性を保つため301リダイレクトを設定。

9-1. 切替手順

  1. microCMS → Supabase の移行スクリプト（scripts/migrate-from-microcms.ts）を作成。各レコードに slug を生成して保存。id（microCMSのcontentId）も hotel_sources に source_type='manual' + source_id=<contentId> として保存。
  2. /hotel/[slug]/page.tsx を新規実装（v2の8-3）
  3. /hotel/[id]/page.tsx をリダイレクトプロキシに改修：
// src/app/hotel/[id]/page.tsx — 移行期間中はリダイレクト
import { redirect, notFound } from 'next/navigation';
import { createServerClient } from '@/lib/supabase/server';

type Props = { params: Promise<{ id: string }> };

export default async function LegacyHotelPage({ params }: Props) {
  const { id } = await params;
  const sb = createServerClient();

  // microCMS時代のidからslugを引く
  const { data } = await sb
    .from('hotel_sources')
    .select('hotel:hotels(slug, status)')
    .eq('source_type', 'manual')

301（永続移転）にしたい場合は next.config.js 側で redirects() を使うか、middleware で 308（Permanent Redirect）にする。SEO的には301／308 が望ましい。

// src/middleware.ts
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
import { createServerClient } from '@/lib/supabase/server';

export async function middleware(req: NextRequest) {
  const match = req.nextUrl.pathname.match(/^\/hotel\/([^\/]+)$/);
  if (!match) return NextResponse.next();
  const seg = match[1];

  // すでにslug形式（英数ハイフン）なら通す
  if (/^[a-z0-9\-]+$/.test(seg)) return NextResponse.next();

  // それ以外（microCMSのcontentId等）はSupabaseで引いて301
  const sb = createServerClient();
  const { data } = await sb
    .from('hotel_sources')
    .select('hotel:hotels(slug)')
    .eq('source_id', seg)
    .maybeSingle();

  if (data?.hotel?.slug) {
    const url = req.nextUrl.clone();
    url.pathname = `/hotel/${data.hotel.slug}`;
    return NextResponse.redirect(url, 301);
  }

  return NextResponse.next();
}

export const config = { matcher: ['/hotel/:path*'] };
注意: middlewareでDBアクセスはレイテンシ増。matcherを /hotel/:path* に絞っているので影響範囲は限定的だが、Vercel Edge Runtime ではsupabase-jsのfetch互換動作要確認。
